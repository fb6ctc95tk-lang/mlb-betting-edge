# Oracle Evaluation Standard

**Edge Oracle — MLB Betting Edge Project**
**Authority Level: 2 — Derives Authority from Oracle Constitution**
**Status: Active**

---

## Preamble

This document establishes the official framework by which every Edge Oracle recommendation is evaluated. It derives its authority from the Oracle Constitution and may expand upon constitutional principles. It may never contradict them.

The Evaluation Standard exists to answer one question with discipline and consistency: was this a good decision, regardless of what happened?

This document governs evaluation methodology only. It does not define betting strategy, handicapping methods, or market-specific play criteria. Those belong in Market Playbooks, which sit below this document in the authority hierarchy.

---

## Section 1 — Purpose

### 1.1 Why This Document Exists

Oracle recommendations are made under uncertainty. Any individual outcome — win or loss — tells an incomplete story about whether the decision that produced it was sound. A sound decision made repeatedly over time produces better outcomes than an unsound one, but no single result confirms or denies this.

The Evaluation Standard exists to build a body of evidence about decision quality that is independent of short-term results. It creates a consistent, auditable record of how well the Oracle reasoned before each game — a record that can be examined honestly, reviewed at milestone intervals, and used to improve the Oracle's process without reacting to variance.

### 1.2 Objectives

This standard pursues four objectives:

1. **Separate process from outcome.** Every recommendation receives a process evaluation that is completed before the game and fixed at first pitch. Outcome is recorded separately and has no retroactive effect on the process evaluation.

2. **Make evaluation consistent.** The same evaluation categories and grading standards apply to every recommendation, regardless of sport, market, confidence level, or result.

3. **Make evaluation auditable.** Every process grade must be supportable by documented pregame reasoning. A grade that cannot be explained by reference to the pregame record is not a valid grade.

4. **Create a foundation for improvement.** Aggregated evaluation data — collected honestly over a meaningful sample — provides the basis for structured milestone reviews and evidence-based governance amendments.

### 1.3 What This Document Does Not Govern

This standard does not:

- Define how to evaluate specific sports, leagues, or game types.
- Specify what data sources constitute valid evidence for any market.
- Define play criteria, entry thresholds, or selection filters for any market.
- Determine what makes a thesis analytically valid within a specific sport.

Those functions belong in Market Playbooks. This standard is intentionally market-agnostic and applies without modification to any sport or market the Oracle may address.

### 1.4 Relationship to the Constitution

This document is subordinate to the Oracle Constitution. Where the Constitution establishes a principle or rule, this standard implements that principle or rule in the context of evaluation. Where this standard is silent on a matter of governance, the Constitution governs.

## Section 2 — Evaluation Philosophy

The following philosophical commitments underlie every element of this standard. They are consistent with and derived from the Founding Principles of the Oracle Constitution.

**A winning recommendation is not automatically a good one.**
A recommendation that wins because the game played out differently than anticipated, or because of a factor the Oracle did not identify, is a beneficiary of variance — not evidence of strong process. The Oracle does not treat winning as validation.

**A losing recommendation is not automatically a poor one.**
A recommendation supported by strong evidence, a sound thesis, a well-chosen market, and fair price that loses because of an unpredictable event reflects good process and bad variance. The Oracle does not treat losing as indictment.

**Pregame reasoning is evaluated before the outcome is known.**
The process evaluation is completed at the time of the recommendation — or immediately before first pitch — and is locked. It is not revised, refined, or softened after the result is known. What the Oracle believed before the game is what it is graded on.

**Long-term process quality is more valuable than short-term variance.**
A consistent, well-graded process applied to a large sample of recommendations produces evidence the Oracle can learn from. A process that shifts in response to recent results produces noise. This standard is designed for the long run.

**Honest evaluation serves the Oracle's purpose. Generous evaluation undermines it.**
Inflating process grades after a win, or deflating them after a loss, destroys the integrity of the evaluation record. The Evaluation Standard is a tool for learning, not for score management. Grades must reflect what the pregame record actually shows, not what the result suggests it should have shown.

## Section 3 — Evaluation Lifecycle

Every Oracle recommendation moves through the following lifecycle. Each stage is distinct. No stage may be performed out of sequence, and no earlier stage may be retroactively altered by a later one.

```
Stage 1 — Recommendation Created
  The Oracle identifies an opportunity and creates a tracker entry.
  Pregame fields are completed: thesis, market, selection, price,
  sportsbook, supporting evidence, risk factors, game script expectation,
  classification, and conditional requirements.

            ↓

Stage 2 — Pregame Process Evaluation
  Before first pitch, the Oracle evaluates its own recommendation
  across the five pregame categories defined in Section 4.
  A Process Score is assigned using the grading framework in Section 6.
  All pregame fields and grades are locked at first pitch.

            ↓

Stage 3 — Game Played
  The game occurs. No Oracle evaluation activity takes place during
  the game. No pregame field or grade may be altered once the game begins.

            ↓

Stage 4 — Outcome Recorded
  The factual result is entered: Win, Loss, Push, Void, or No Play.
  Closing line and closing price are recorded if available.
  Mock profit or loss is calculated per the Tracking Rules.

            ↓

Stage 5 — Postgame Review
  The Oracle completes the postgame evaluation defined in Section 5:
  closing-line value, game-script accuracy, and variance classification.
  Postgame fields are additive only. They do not alter pregame fields.

            ↓

Stage 6 — Variance Assessment
  The Oracle classifies the relationship between the process quality,
  the game-script accuracy, and the result using the Variance
  Classification framework in Section 8.

            ↓

Stage 7 — Lessons Learned
  The Oracle classifies any lesson from the recommendation using
  the framework in Section 9. The lesson classification is advisory.
  It does not modify any governance document.

            ↓

Stage 8 — Tracker Updated
  All postgame fields are finalized and saved to the tracker record.
  The record is now complete. No field in the record may be altered
  after this stage except through the Constitutional Amendment Procedure.

            ↓

Stage 9 — Future Audit
  The complete record is available for milestone review at the
  intervals defined in the Tracking Rules. Aggregated evaluation
  data informs the audit process. Audit findings may initiate
  the Constitutional Amendment Procedure.
```

## Section 4 — Pregame Process Evaluation

Before first pitch, the Oracle evaluates every active recommendation across five categories. Each category assesses a distinct dimension of decision quality. Each is graded independently using the Process Score framework in Section 6.

The five categories are:

1. Game Selection
2. Market Selection
3. Handicap Quality
4. Value and Price
5. Execution and Discipline

No category is weighted above another in principle. The Process Score reflects the overall quality of the recommendation as a whole, informed by the performance across all five categories.

---

### 4.1 Game Selection

**What is being evaluated:**
Whether this was the right opportunity to engage. Game Selection assesses whether the Oracle identified a genuine analytical edge in this specific matchup on this specific date — and whether the available evidence was sufficient to justify a recommendation rather than a pass.

**Why it matters:**
The Oracle's value comes partly from recognizing which games to play and which to avoid. Engaging in a game with insufficient evidence, no meaningful analytical distinction between the two sides, or no clear thesis degrades the quality of the record regardless of the result. Passing is always a valid decision. Playing for the sake of activity is not.

**Strong Game Selection:**
- A specific, documentable reason exists to engage this game on this date.
- The analytical picture is clear enough to support a directional thesis.
- The Oracle can explain why this game, rather than others on the same slate, merits a recommendation.
- Passing on adjacent games with weaker evidence is consistent with the selection.

**Weak Game Selection:**
- No specific reason distinguishes this game from others on the slate.
- The thesis is generic and would apply equally to many games.
- The recommendation is made because it is a notable game, not because an edge exists.
- The Oracle is playing to fill a quota rather than to identify a genuine opportunity.

---

### 4.2 Market Selection

**What is being evaluated:**
Whether the chosen market is the correct vehicle for the identified thesis. Market Selection assesses whether the Oracle selected the bet type — moneyline, run line, total, first-half, player prop, or other — that best represents its analytical position.

**Why it matters:**
A strong thesis expressed in the wrong market may fail to capture the identified edge, may expose the recommendation to risks the thesis does not address, or may offer a worse price profile than an alternative market that better aligns with what the Oracle believes. Choosing the right market amplifies the value of a correct analysis. Choosing the wrong one wastes it.

**Strong Market Selection:**
- The chosen market directly reflects the Oracle's thesis.
- The Oracle can explain why this market and not an alternative captures the identified edge.
- The risk profile of the market is consistent with the nature of the analytical case.

**Weak Market Selection:**
- The thesis supports one position but the Oracle plays a different market without justification.
- The chosen market exposes the recommendation to additional risks the thesis does not address.
- The market was selected for pricing convenience rather than analytical alignment.

---

### 4.3 Handicap Quality

**What is being evaluated:**
The quality of the analytical work behind the recommendation. Handicap Quality assesses whether the Oracle's thesis is built on credible, specific, relevant evidence; whether the reasoning is logically sound; and whether the Oracle has identified and weighed the arguments against its position.

**Why it matters:**
This is the intellectual core of every recommendation. A thesis built on weak evidence, incomplete research, or circular reasoning will not hold up across a large sample, regardless of early results. The Oracle's long-term value depends on consistently applying rigorous analytical standards.

This category does not evaluate whether the analysis was correct. It evaluates whether it was sound — whether the available information was used thoughtfully, completely, and honestly.

**Strong Handicap Quality:**
- Multiple independent factors support the thesis direction.
- The Oracle has considered and addressed the strongest counterarguments.
- The evidence cited is specific: named data points, observed trends, confirmed conditions.
- The thesis is falsifiable — the Oracle can identify what would change its view.
- Risk factors are honestly assessed and documented.

**Weak Handicap Quality:**
- The thesis rests on a single factor or on general narrative.
- Counterarguments are ignored or dismissed without engagement.
- Evidence is vague, secondary, or unverifiable.
- The Oracle cannot state what evidence would have produced a pass.
- Risk factors are omitted or understated.

---

### 4.4 Value and Price

**What is being evaluated:**
Whether the entry price represents fair or favorable compensation for the risk being accepted. Value and Price assesses whether the odds at which the recommendation is made are consistent with the Oracle's assessment of the true probability of the outcome, and whether the price is worth accepting given the quality of the thesis.

**Why it matters:**
Even a well-constructed thesis becomes a poor recommendation if entered at a price that undercompensates for the risk. Conversely, a modestly constructed thesis may offer positive value at the right price. This category ensures the Oracle does not treat any identified analytical edge as sufficient justification for any price.

This category does not require a precise probability calculation or a fair-odds model. It requires an honest assessment of whether the available price is consistent with the Oracle's analytical view.

**Strong Value and Price:**
- The Oracle can articulate why the available price is acceptable or favorable given the thesis.
- The price is consistent with or better than what the Oracle believes the true probability implies.
- Any price condition was verified and confirmed before the recommendation was activated.

**Weak Value and Price:**
- The Oracle accepted an unfavorable price without justification.
- The entry price is significantly worse than what the thesis implies is fair.
- A price condition was set but accepted below the stated threshold.
- The Oracle ignored the price entirely and focused only on the directional thesis.

---

### 4.5 Execution and Discipline

**What is being evaluated:**
Whether the recommendation was made and documented in compliance with all applicable Constitutional Rules and Tracking Rules. Execution and Discipline assesses the operational integrity of the recommendation — not its analytical quality, but whether the process was followed correctly.

**Why it matters:**
Analytical quality means nothing if the Oracle does not apply it consistently. Forced plays, oversized stakes, skipped documentation, and ignored conditional gates all undermine the integrity of the evaluation record. This category ensures that each recommendation reflects both sound analysis and correct process.

**Strong Execution and Discipline:**
- The recommendation falls within the daily play limit.
- Flat 1-unit paper stake is applied without deviation.
- All required pregame fields are completed before first pitch.
- Any conditional requirements were explicitly confirmed before activation.
- The Oracle did not force a play in the absence of qualifying evidence.

**Weak Execution and Discipline:**
- A Constitutional Rule was violated in the making or recording of the recommendation.
- Required documentation was incomplete at first pitch.
- A conditional play was activated without confirmation of the stated conditions.
- Play volume on the day exceeded the permitted limit.
- A pass was overridden without documented justification.

## Section 5 — Postgame Evaluation

Postgame evaluation is performed after the game ends. Every postgame field is additive: it records what happened against what was predicted. No postgame field alters any pregame field or pregame grade.

### 5.1 Result

The factual outcome of the recommendation. Entered after the game ends.

Permitted values: Win, Loss, Push, Void, No Play.

The Result field is a fact, not an evaluation. It has no retroactive effect on the Process Score or on any pregame category grade. See Section 7 for outcome definitions.

### 5.2 Closing Line Value

Closing Line Value (CLV) records the relationship between the price at which the recommendation was entered and the price at which the market closed immediately before game start.

CLV is expressed as a directional comparison:
- **Beat the Close:** The entry price was more favorable than the closing price.
- **Tied the Close:** The entry price and closing price were equivalent.
- **Lost to the Close:** The entry price was less favorable than the closing price.

CLV is an observation, not a grade. It reflects whether the market moved in favor of or against the Oracle's position between entry and close. Over a large sample, consistently beating the close is evidence that the Oracle is identifying value before the market fully prices it in.

CLV cannot be calculated for recommendations where no closing price is available. In those cases, the field is marked unavailable. This does not invalidate the recommendation.

The methodology for calculating CLV for specific market types is a market-level detail. It belongs in the relevant Market Playbook, not in this standard.

### 5.3 Game Script Accuracy

Game Script Accuracy records how closely the actual game resembled the expected game script documented in the pregame record.

This field contains two components:
- **Actual Game Script:** A brief factual description of how the game played out.
- **Script Accuracy:** A qualitative assessment comparing the actual game to the expected script.

Permitted values for Script Accuracy: As Expected, Partially As Expected, Diverged.

Script Accuracy is independent of the Result. A recommendation that wins because the game followed an unexpected path may receive a low Script Accuracy assessment. A recommendation that loses in a game that played out exactly as predicted receives a high one.

Script Accuracy informs the Variance Classification in Section 5.4.

### 5.4 Variance Classification

Variance Classification describes the relationship between the Oracle's pregame analytical model, the game script that actually occurred, and the result. See Section 8 for complete definitions.

The classification is assigned after Script Accuracy is determined.

### 5.5 Repeat Decision Assessment

The Repeat Decision Assessment records whether, under the same pregame conditions, the Oracle would make the same recommendation again.

Permitted values: Yes, No, Conditional.

**Yes:** The process was sound. The same recommendation would be made again under the same conditions.

**No:** A meaningful flaw in the process has been identified. The same recommendation would not be made again.

**Conditional:** The recommendation would be made again under modified conditions. The condition must be specified.

The Repeat Decision is based on process quality and conditions — not on the result. A recommendation that lost due to negative variance may receive a Repeat Decision of Yes. A recommendation that won despite weak process may receive a Repeat Decision of No.

The Repeat Decision is advisory. It informs future analysis. It does not modify governance documents or Market Playbooks.

### 5.6 Lessons Learned

Lessons Learned captures the principal analytical or process observation from the completed recommendation. See Section 9 for classification definitions.

## Section 6 — Process Score

Every completed recommendation receives a single Process Score that reflects the overall quality of the pregame decision across all five evaluation categories. The Process Score is assigned after all five pregame category evaluations are complete, before first pitch.

The Process Score uses a five-level letter grade scale.

---

**A — Strong Process**

All five pregame categories are well-executed. The recommendation is built on specific, credible evidence. The market choice clearly reflects the thesis. The price is acceptable. Execution is clean. The Oracle has honestly assessed and documented risk factors and counterarguments. A reader reviewing the pregame record without knowledge of the result would find the decision well-supported.

---

**B — Sound Process**

The recommendation is solid across most categories. One category has a minor weakness that does not undermine the overall case. The thesis is supportable, the market is reasonably well-matched, the price is acceptable, and no Constitutional Rules were violated. A reader would find the decision reasonable, with a specific area that could have been stronger.

---

**C — Acceptable Process**

The recommendation has meaningful strengths and meaningful gaps. The thesis is plausible but partly built on general reasoning or incomplete evidence. At least two categories are weaker than they should be. The decision is defensible but not well-constructed. A reader would find the decision understandable but would identify clear areas where the analysis was insufficient.

---

**D — Weak Process**

The recommendation has more gaps than strengths. The thesis is thin, the market may be poorly matched to the thesis, the price may not be justified, or the process may have been partially compromised. The recommendation was made, but the pregame record does not adequately support it. A reader would find the decision difficult to justify from the available evidence.

---

**F — Failed Process**

The recommendation should not have been made. A Constitutional Rule was violated, required documentation was absent at first pitch, the play was forced without qualifying evidence, or the analytical case is so weak as to constitute no case at all. A reader reviewing the pregame record would find no adequate basis for the recommendation. An F grade does not void the recommendation or its result — it records that the process failed.

---

**Assigning the Process Score:**

The Process Score is a holistic judgment, not a mechanical average. A recommendation with four strong categories and one Constitutional Rule violation receives an F, not a B. A recommendation with three B-level categories and two A-level categories receives an A or B depending on how the overall record reads. The Oracle exercises judgment in assigning the Process Score and must be able to explain it by reference to the five category evaluations.

---

## Section 7 — Outcome Classification

Outcome is a factual record. It does not determine process quality. The Outcome Classification records what happened; the Process Score records how well the Oracle reasoned before it did.

---

**Win**
The selected outcome occurred. Profit is calculated per the applicable Tracking Rule.

**Loss**
The selected outcome did not occur. Loss is recorded as one unit.

**Push**
The outcome resulted in a tie on the selected market. No profit or loss is recorded.

**Void**
The recommendation was conditionally active and the required condition was not confirmed before first pitch. A Void is not a loss. It is an accurate application of the conditional play rules. Profit and loss are recorded as zero.

**No Play**
The recommendation was entered in the tracker but was not activated — either because the Oracle passed upon review of final pregame conditions, or because the game was postponed, cancelled, or otherwise did not occur under circumstances outside the Oracle's analytical scope. Profit and loss are recorded as zero.

---

Outcome Classification never determines Process Score.
Process Score never determines Outcome Classification.
These two fields exist in separate columns and are evaluated independently.

---

## Section 8 — Variance Classification

Variance Classification describes the relationship between what the Oracle predicted and what occurred. It is assigned after the game ends, using the pregame record, the Script Accuracy assessment, and the result.

Variance Classification is one of four values:

---

**Expected**

The game played out in a manner consistent with the pregame thesis and expected game script. The result — win or loss — is what a reasonable reading of the pregame record would have anticipated. The Oracle's model described the game reasonably well.

An Expected classification combined with a Win confirms the process. An Expected classification combined with a Loss is informative: the thesis was correct in structure but did not produce the anticipated outcome. This is the most important variance category for long-term learning.

---

**Positive Variance**

The result was better than the pregame model predicted, and the improvement came from a factor the Oracle did not anticipate. The game may have diverged from the expected script in the Oracle's favor, or an unexpected event produced a favorable outcome.

Positive Variance combined with a Win is a cautionary signal: the Oracle should not treat the win as confirmation of its process if the result was driven by factors outside the thesis. Positive Variance combined with a Push or No Play is neutral.

---

**Negative Variance**

The result was worse than the pregame model predicted, and the shortfall came from a factor the Oracle did not anticipate. The game may have diverged from the expected script against the Oracle, or an unexpected event produced an unfavorable outcome.

Negative Variance combined with a Loss should not trigger a process review unless the factor that caused the variance was something the Oracle should have identified. If the variance factor was genuinely unpredictable, the loss is an honest cost of operating under uncertainty, not evidence of a process failure.

---

**Model Miss**

The Oracle's analytical model was structurally wrong in a way that reflects a gap in the evaluation framework itself — not simply bad luck or an unpredictable event. A Model Miss occurs when the Oracle's thesis was directionally flawed or based on a category of evidence that did not apply as assumed.

A Model Miss does not necessarily produce an F Process Score. The Oracle may have followed a sound process and arrived at a wrong structural conclusion. However, a Model Miss is the most important variance category for governance purposes: multiple Model Miss classifications of the same type, across different plays, are the strongest signal that a methodology review is warranted. These should be flagged in milestone Audit Reports.

## Section 9 — Lessons Learned

Every completed recommendation produces a Lessons Learned classification. The classification is assigned after Variance is determined. It is advisory and informational. It does not modify any governance document, Market Playbook, or Tracking Rule.

---

**No Action Required**

The recommendation was well-constructed and the result — whatever it was — did not reveal a gap in process or methodology. The record is complete and requires no follow-up. A No Action Required classification on a loss with an Expected variance and a strong Process Score is a sign of a healthy evaluation record.

---

**Monitor**

A pattern or observation worth noting has appeared, but it has been seen too few times to draw conclusions. The lesson is documented and the tracker entry is flagged for inclusion in the next milestone review. No change to any governance document is proposed at this stage. The Monitor classification exists to build awareness without reacting prematurely.

---

**Investigate**

A specific, identifiable gap in the analytical process has been observed. The gap is documentable by reference to the pregame record. The Oracle flags the entry for structured review at the next milestone, with a note describing the specific gap and which evaluation category it touches. An Investigate classification may lead to a governance proposal at the appropriate milestone — it does not trigger one automatically.

---

**Recommend Governance Review**

A structural issue has been identified that appears to reflect a limitation or gap in the Oracle's governing documents — not merely a one-time analytical error. This classification is used sparingly and only when the Oracle can clearly articulate what governance document, principle, or rule the issue touches. A Recommend Governance Review classification does not initiate the Constitutional Amendment Procedure. It creates a documented record that, when it appears multiple times across a meaningful sample, provides the evidence base for a formal Amendment Proposal under Article VI of the Constitution.

---

Lessons Learned classifications are inputs to milestone reviews. They are never outputs of them. A milestone review produces Audit Reports and, where evidence warrants, Amendment Proposals. It does not produce retroactive changes to Lessons Learned classifications.

## Section 10 — Governance Rules

The following rules govern the use and integrity of the Evaluation Standard. They derive from the Oracle Constitution and may not be modified except through the Constitutional Amendment Procedure.

**Rule 1 — Evaluation May Not Modify the Constitution**
No finding, grade, classification, or lesson produced under this standard may directly amend, suspend, or override any provision of the Oracle Constitution. Amendments to the Constitution require the full eight-stage procedure defined in Article VI of the Constitution.

**Rule 2 — Evaluation May Not Modify Market Playbooks**
A process grade, variance classification, or Lessons Learned entry does not constitute authorization to change any Market Playbook. Market Playbook amendments follow the same Constitutional Amendment Procedure. A single evaluation finding — regardless of how clearly it identifies a market-specific gap — is an observation, not a mandate.

**Rule 3 — Evaluation May Not Rewrite History**
Once a pregame field or pregame grade has been completed and locked at first pitch, no evaluation activity — including this standard — may alter it. The postgame record is additive only. A poor result that prompts a desire to revise the pregame thesis does not constitute grounds for revision.

**Rule 4 — Evaluation May Not Change Tracker Records Retroactively**
Tracker records are finalized at Stage 8 of the Evaluation Lifecycle. No correction, reclassification, or update to a completed tracker record is permitted except to correct a factual error — such as a miscalculated profit figure — that is documented as a correction and distinguished from a substantive revision. Factual corrections must be noted in Review Notes with an explanation.

**Rule 5 — Evaluation Follows Constitutional Governance**
All activities conducted under this standard — grading, classification, lesson assignment, and milestone review — are subject to the authority hierarchy, founding principles, and constitutional rules established in the Oracle Constitution. Where this standard is silent or ambiguous, the Constitution governs.

**Rule 6 — Process Scores Are Assigned Before First Pitch**
The Process Score and all five pregame category evaluations are completed before the game begins. They may not be deferred to the postgame period for any reason, including uncertainty about how to grade a category. If a category cannot be evaluated with the available pregame information, the Oracle must make its best judgment before first pitch and document the basis for that judgment.

**Rule 7 — Honest Grading Is Required**
The integrity of the Evaluation Standard depends entirely on the honesty of the grades assigned. The Oracle must grade recommendations as the pregame record actually reads — not as the result suggests they should have read. A Process Score inflated by a win or deflated by a loss is a false record and violates Constitutional Principle III (No Hindsight) and Principle VI (Serves the Process, Not the Ego).

## Future Governance Recommendations (Not Implemented)

The following items were identified during the drafting of this standard. They are outside the approved scope of this sprint and require separate Constitution Authority approval before implementation. They are recorded here for reference only and have no binding effect.

---

**FGR-1 — Grade Anchor Supplement**

Each of the five pregame evaluation categories could benefit from a detailed Grade Anchor document that defines, in specific terms, what an A, B, C, D, and F look like for that category. Such anchors would reduce evaluator subjectivity and improve consistency across a large sample. This would be a supplement to this standard, not a modification of it. Recommended for a future documentation sprint after 30 completed plays provide real examples.

**FGR-2 — Tracker Column Specification**

The Evaluation Standard implies a set of tracker columns not currently present in the paper-bet tracker template. A formal Tracker Update Sprint would be needed to add pregame grade columns (one per evaluation category), Process Score, CLV fields, Script Accuracy, Variance Classification, Repeat Decision, and Lesson Classification to the tracker. This sprint is separate from this document sprint and requires separate authorization.

**FGR-3 — CLV Calculation Methodology by Market Type**

Closing Line Value calculation varies by market. For moneyline bets, CLV is straightforward. For spread and total bets, the spread or number must be accounted for separately from the price. For player props, closing data may be unavailable. The appropriate home for market-specific CLV methodology is within each Market Playbook, not this standard. This recommendation flags the need for Market Playbooks to address CLV explicitly when they are drafted.

**FGR-4 — Milestone Audit Report Template**

The Constitution references milestone reviews at 30 plays, 100 plays, and annually. No formal audit report template currently exists in the repository. A standardized template would ensure that audit findings are documented consistently, include the required evidence (Play IDs, evaluation distributions, variance distributions, Lessons Learned classifications), and produce outputs suitable for informing Constitutional Amendment Proposals. Recommended as a future documentation sprint after the Evaluation Standard is in active use.

**FGR-5 — Numerical Grade Boundaries**

Some evaluators may benefit from explicit numerical guidance mapping Process Score letter grades to a distribution across the five category assessments. For example: an A may require that no category falls below a specified threshold. These boundaries would reduce subjectivity without introducing numerical formulas into the grading act itself. This is advisory and should not be implemented until real evaluation data exists to validate the boundaries.
